1. PROJECT CONTEXT & ARCHITECTURE OVERVIEW
================================================================================

Genius Study Buddy (GSB) is a cloud-based, AI-powered educational SaaS platform
for Nigerian secondary schools. It is multi-tenant: each school is a completely
isolated tenant. One school's data MUST NEVER be accessible to another school.

PLATFORM ENTRY POINTS
──────────────────────
There are exactly TWO entry paths — this must drive the entire navigation
architecture:

  Path A — VISITOR:
    Landing → Visitor Registration → Email Verification → Visitor Dashboard

  Path B — SCHOOL USER:
    Landing → School Selection → Role Selection → Login → Role Dashboard

FIVE USER ROLES
───────────────
  STUDENT    — authenticates via admission number (username), no email required
  TEACHER    — authenticates via email
  MAIN_ADMIN — authenticates via email, highest school authority
  SUB_ADMIN  — authenticates via email, limited admin
  VISITOR    — authenticates via email, 1-week trial, no school affiliation

TENANT ISOLATION MECHANISM
───────────────────────────
The backend embeds school_id and role inside the JWT payload. TenantMiddleware
reads school_id from the JWT and executes:

  SET LOCAL app.current_school_id = '<uuid>';

This runs a PostgreSQL Row-Level Security (RLS) policy at the DB kernel level.
The frontend must:
  1. Never send requests to another school's resources
  2. Always pass the Authorization: Bearer <access_token> header
  3. Decode the JWT to read role and school_id for client-side routing decisions

ASYNC PIPELINE (critical for Phase 2 UX)
──────────────────────────────────────────
Note processing is entirely asynchronous via Celery workers. The frontend polls
lightweight status endpoints until the pipeline completes. Three pipelines exist:

  OCR Pipeline:    PENDING_OCR → AWAITING_STUDENT_APPROVAL → (student confirms) → PROCESSING_AI → READY | FAILED
  AI Pipeline:     PROCESSING_AI → READY | FAILED
  Conformity:      PENDING → PROCESSING → DONE | FAILED

Polling MUST stop when status reaches a terminal state (READY, DONE, FAILED or
AWAITING_STUDENT_APPROVAL which requires user action).

3. ENVIRONMENT CONFIGURATION
================================================================================

Create .env.local (never commit this):

  NEXT_PUBLIC_API_BASE_URL=http://localhost:8000
  NEXT_PUBLIC_APP_URL=http://localhost:3000

Create .env.production:

  NEXT_PUBLIC_API_BASE_URL=https://api.yourdomain.com
  NEXT_PUBLIC_APP_URL=https://yourdomain.com

IMPORTANT: Only variables prefixed NEXT_PUBLIC_ are exposed to the browser.
The API base URL is public because the frontend calls it from the browser.
Never put secret keys in NEXT_PUBLIC_ variables.

Access in code:
  const API_BASE = process.env.NEXT_PUBLIC_API_BASE_URL!;

5. COMPLETE API CONTRACT — PHASE 1 & PHASE 2
================================================================================

BASE URL:  process.env.NEXT_PUBLIC_API_BASE_URL + "/api/v1"
Example:   http://localhost:8000/api/v1

All authenticated endpoints require:
  Header: Authorization: Bearer <access_token>
  Header: Content-Type: application/json  (except multipart uploads)

PAGINATION — Standard response shape for all paginated endpoints:
  {
    count:       number,   // total items
    next:        string | null,
    previous:    string | null,
    total_pages: number,
    results:     T[]
  }

ERROR RESPONSE — RFC 7807 format (ALL errors from the backend):
  {
    error_code:  string,   // machine-readable code e.g. "INVALID_CREDENTIALS"
    detail:      string,   // human-readable message
    status_code: number,
    timestamp:   string    // ISO 8601
  }

DRF validation errors (400) have this shape instead:
  {
    field_name: ["error message"],
    non_field_errors: ["error message"]
  }

5.2 AUTHENTICATION ENDPOINTS
──────────────────────────────────────────────────────────────────────────────

POST /auth/login/
  Purpose: Dual-vector login for all roles.
  Auth: None. Rate-limited: 5/minute per IP.
  Request:
    {
      role:       "STUDENT" | "TEACHER" | "MAIN_ADMIN" | "SUB_ADMIN" | "VISITOR"
      identifier: string   // admission number (STUDENT) or email (others)
      password:   string
      school_id:  string   // UUID — REQUIRED for all roles EXCEPT VISITOR
    }
  Response 200:
    {
      access:  string   // JWT, expires 15 minutes
      refresh: string   // JWT, expires 7 days
    }
  Errors:
    401 INVALID_CREDENTIALS   — wrong username/password
    403 ACCOUNT_DISABLED      — is_active = false
    403 ROLE_MISMATCH          — role param doesn't match stored role
    403 EMAIL_NOT_VERIFIED     — VISITOR who hasn't verified email

  JWT PAYLOAD (decoded from access token):
    {
      user_id:   string (UUID)
      role:      string
      school_id: string | null   // null for VISITOR
      exp:       number
      iat:       number
    }

POST /auth/token/refresh/
  Purpose: Get a new access token using the refresh token.
  Auth: None.
  Request:  { refresh: string }
  Response 200: { access: string }
  Error:
    401 — expired or blacklisted refresh token → user must re-login

POST /auth/register/visitor/
  Purpose: Self-service visitor account creation.
  Auth: None. Rate-limited: 5/minute.
  Request:
    {
      email:      string
      password:   string
      first_name: string
      last_name:  string
    }
  Response 201:
    { detail: "Account created. Please check your email to verify your address." }
  Errors:
    400 — validation errors (email already in use, weak password, etc.)

POST /auth/verify-email/
  Purpose: Complete visitor email verification. Token comes from email link.
  Auth: None.
  Request:  { token: string }  // UUID from ?token= query param
  Response 200:
    {
      detail:  string
      access:  string   // immediately logged in after verification
      refresh: string
    }
  Errors:
    400 INVALID_TOKEN  — token not found
    400 TOKEN_EXPIRED  — token expired (24h TTL) — prompt to resend

POST /auth/resend-verification/
  Purpose: Re-send verification email. Always returns 200 (no enumeration).
  Auth: None. Rate-limited: 5/minute.
  Request:  { email: string }
  Response 200: { detail: string }

POST /auth/logout/
  Purpose: Blacklist the refresh token.
  Auth: Required.
  Request:  { refresh: string }
  Response 204: (no body)
  Error:
    400 MISSING_TOKEN / INVALID_TOKEN

GET /auth/me/
  Purpose: Fetch the authenticated user's own profile.
  Auth: Required.
  Response 200:
    {
      id:                string
      email:             string | null
      username:          string | null    // student admission number
      role:              string
      school:            string | null    // school UUID
      first_name:        string
      last_name:         string
      is_email_verified: boolean
      trial_expires_at:  string | null    // ISO 8601, visitors only
      date_joined:       string
    }

PATCH /auth/me/
  Purpose: Update own display name.
  Auth: Required.
  Request:  { first_name?: string, last_name?: string }
  Response 200: (same shape as GET /auth/me/)

POST /auth/password/change/
  Purpose: Change own password (authenticated).
  Auth: Required.
  Request:
    {
      old_password:     string
      new_password:     string
      confirm_password: string
    }
  Response 204
  Errors:
    400 WRONG_PASSWORD     — old_password is incorrect
    400 — validation (password too short, doesn't match, etc.)

POST /auth/password/reset/
  Purpose: Trigger a password reset email. Always 200 (no enumeration).
  Auth: None.
  Request:  { email: string }
  Response 200: { detail: string }

POST /auth/password/reset/confirm/
  Purpose: Set new password using uid+token from reset email.
  Auth: None.
  Request:
    {
      uid:              string   // base64url-encoded user pk
      token:            string   // Django PasswordResetTokenGenerator token
      new_password:     string
      confirm_password: string
    }
  Response 204
  Errors:
    400 INVALID_RESET_LINK — expired or invalid uid/token

  Response 200: SchoolCulture object (same shape as /schools/{id}/culture/)
  NOTE: Returns empty strings if culture hasn't been configured yet.

PUT /admin/culture/
  NOTE: This is a full upsert — use PUT, not PATCH. Send ALL fields.
  Request:
    {
      motto?:             string
      mission_statement?: string
      vision_statement?:  string
      core_values?:       string
      school_creed?:      string
      founder_quotes?:    Array<{ quote: string, author: string }>

  Response 200: Updated SchoolCulture object

import axios, { AxiosError, InternalAxiosRequestConfig } from 'axios';
import { getTokens, setAccessToken, clearTokens } from '@/lib/token';

const API_BASE = process.env.NEXT_PUBLIC_API_BASE_URL + '/api/v1';

export const apiClient = axios.create({
  baseURL: API_BASE,
  headers: { 'Content-Type': 'application/json' },
  timeout: 30_000,
});

// ── Request interceptor — inject Bearer token ─────────────────────────────
apiClient.interceptors.request.use((config: InternalAxiosRequestConfig) => {
  const { access } = getTokens();
  if (access) {
    config.headers.Authorization = `Bearer ${access}`;
  }
  return config;
});

// ── Response interceptor — silent token refresh ───────────────────────────
let isRefreshing = false;
let failedQueue: Array<{
  resolve: (token: string) => void;
  reject:  (err: unknown) => void;
}> = [];

const processQueue = (error: unknown, token: string | null) => {
  failedQueue.forEach(({ resolve, reject }) =>
    error ? reject(error) : resolve(token!)
  );
  failedQueue = [];
};

apiClient.interceptors.response.use(
  (response) => response,
  async (error: AxiosError) => {
    const originalRequest = error.config as InternalAxiosRequestConfig & { _retry?: boolean };

    if (error.response?.status === 401 && !originalRequest._retry) {
      if (isRefreshing) {
        return new Promise((resolve, reject) => {
          failedQueue.push({ resolve, reject });
        }).then((token) => {
          originalRequest.headers.Authorization = `Bearer ${token}`;
          return apiClient(originalRequest);
        });
      }

      originalRequest._retry = true;
      isRefreshing = true;

      const { refresh } = getTokens();
      if (!refresh) {
        clearTokens();
        window.location.href = '/';
        return Promise.reject(error);
      }

      try {
        const { data } = await axios.post(`${API_BASE}/auth/token/refresh/`, { refresh });
        setAccessToken(data.access);
        processQueue(null, data.access);
        originalRequest.headers.Authorization = `Bearer ${data.access}`;
        return apiClient(originalRequest);
      } catch (refreshError) {
        processQueue(refreshError, null);
        clearTokens();
        window.location.href = '/?session=expired';
        return Promise.reject(refreshError);
      } finally {
        isRefreshing = false;
      }
    }

    return Promise.reject(error);
  }
);

const ACCESS_KEY  = 'gsb_access';
const REFRESH_KEY = 'gsb_refresh';

// SECURITY NOTE: Use sessionStorage (clears on tab close) for access tokens.
// For a higher-security deployment, store tokens in httpOnly cookies via a
// Next.js API route acting as a BFF proxy. For the MVP, sessionStorage is
// acceptable but document the tradeoff.

export const getTokens = () => ({
  access:  sessionStorage.getItem(ACCESS_KEY)  ?? '',
  refresh: localStorage.getItem(REFRESH_KEY)   ?? '',
});

export const setTokens = (access: string, refresh: string) => {
  sessionStorage.setItem(ACCESS_KEY,  access);
  localStorage.setItem(REFRESH_KEY,   refresh);
};

export const setAccessToken = (access: string) => {
  sessionStorage.setItem(ACCESS_KEY, access);
};

export const clearTokens = () => {
  sessionStorage.removeItem(ACCESS_KEY);
  localStorage.removeItem(REFRESH_KEY);
};

// Decode JWT payload WITHOUT verification (verification is the server's job).
export const decodeToken = (token: string): JWTPayload | null => {
  try {
    const { jwtDecode } = await import('jwt-decode');
    return jwtDecode<JWTPayload>(token);
  } catch {
    return null;
  }
};

// Use the synchronous version:
import { jwtDecode } from 'jwt-decode';
import type { JWTPayload } from '@/types';

  catch { return null; }

8. AUTHENTICATION ARCHITECTURE & TOKEN STRATEGY
================================================================================

PRE-LOGIN STATE MACHINE
────────────────────────
The login flow has a specific multi-step sequence that must be maintained:

  Step 1: User arrives at landing page.
  Step 2: User chooses "Enter as Visitor" OR "Enter Through School".

  PATH A — Visitor:
    → /register (visitor registration form)
    → Check email → /verify-email?token=<uuid>
    → Verified → tokens issued → /visitor/dashboard

  PATH B — School User:
    → /schools (paginated school grid, search bar)
    → Click school card → store school in Zustand (schoolStore)
    → /login?school_id=<uuid>&school_name=<name>
    → Role selection (Student / Teacher / Admin tabs on login page)
    → Submit credentials → tokens issued → role-specific dashboard

TOKEN STORAGE STRATEGY
───────────────────────
  access token  → sessionStorage (automatically cleared when browser tab closes)
  refresh token → localStorage   (persists across browser sessions for 7 days)

On app load:
  1. Read access token from sessionStorage.
  2. If missing but refresh exists in localStorage → call /auth/token/refresh/
     silently before rendering protected pages.
  3. If both missing → user is unauthenticated → redirect to landing.

ZUSTAND AUTH STORE
───────────────────
On successful login:
  1. Call setTokens(access, refresh)
  2. Decode access token → extract { user_id, role, school_id }
  3. Store decoded payload in authStore
  4. Redirect based on role:
       STUDENT    → /student/dashboard
       TEACHER    → /teacher/dashboard
       MAIN_ADMIN → /admin/dashboard
       SUB_ADMIN  → /admin/dashboard
       VISITOR    → /visitor/dashboard
       is_staff   → /platform/dashboard  (check this AFTER decoding JWT)

ROUTE PROTECTION STRATEGY
───────────────────────────
Use Next.js middleware (src/middleware.ts) for server-side token existence checks.
Use RoleGuard component for client-side role checks within route groups.

src/middleware.ts skeleton:
  import { NextResponse } from 'next/server';
  import type { NextRequest } from 'next/server';

  const PUBLIC_PATHS = ['/', '/schools', '/login', '/register',
                        '/verify-email', '/forgot-password', '/reset-password'];

  export function middleware(request: NextRequest) {
    const { pathname } = request.nextUrl;
    const isPublic = PUBLIC_PATHS.some(p => pathname.startsWith(p));
    const refreshToken = request.cookies.get('gsb_refresh')?.value;
    // If using cookies — redirect to login if no refresh token on protected path
    if (!isPublic && !refreshToken) {
      return NextResponse.redirect(new URL('/', request.url));
    }
    return NextResponse.next();
  }

NOTE: If storing tokens in localStorage/sessionStorage (not cookies), middleware
cannot read them. In that case, use client-side RoleGuard in each layout.tsx.

interface AuthState {

  refreshToken: string | null;
  userId:       string | null;
  role:         UserRole | null;
  schoolId:     string | null;
  isStaff:      boolean;
  isAuthenticated: boolean;

export const useAuthStore = create<AuthState>()(
  persist(
    (set) => ({
      accessToken:     null,
      refreshToken:    null,
      userId:          null,
      role:            null,
      schoolId:        null,
      isStaff:         false,
      isAuthenticated: false,

      setTokens: (access, refresh, payload) => set({
        accessToken:     access,
        refreshToken:    refresh,
        userId:          payload.user_id,
        role:            payload.role,
        schoolId:        payload.school_id,
        isStaff:         false, // set separately if is_staff is in payload
        isAuthenticated: true,
      }),

      clearAuth: () => set({
        accessToken: null, refreshToken: null,
        userId: null, role: null, schoolId: null,
        isStaff: false, isAuthenticated: false,
      }),
    }),
    {
      name: 'gsb-auth',
      partialize: (state) => ({
        // Only persist non-sensitive routing state
        role: state.role,
        schoolId: state.schoolId,
        isAuthenticated: state.isAuthenticated,
      }),
    }
  )
);

    defaultOptions: {
      queries: {
        staleTime:            60 * 1000,    // 1 minute
        gcTime:               5 * 60 * 1000, // 5 minutes (formerly cacheTime)
        retry:                1,
        refetchOnWindowFocus: false,
      },
    },
  }));

  return (

    </html>
  );
}

POLLING PATTERN — Note Status
───────────────────────────────
// src/hooks/useNotePoller.ts
import { useQuery } from '@tanstack/react-query';
import { queryKeys } from '@/lib/query-keys';
import { getNoteStatus } from '@/lib/api/notes';
import type { NoteStatus } from '@/types';

const TERMINAL_STATES: NoteStatus[] = [
  'READY',
  'FAILED',
  'AWAITING_STUDENT_APPROVAL',  // Requires user action — stop auto-polling
];

export function useNotePoller(noteId: string, initialStatus: NoteStatus) {
  return useQuery({
    queryKey: queryKeys.notes.status(noteId),
    queryFn:  () => getNoteStatus(noteId),
    enabled:  !!noteId,
    // Poll every 3 seconds while in an active processing state
    refetchInterval: (query) => {
      const status = query.state.data?.status ?? initialStatus;
      return TERMINAL_STATES.includes(status) ? false : 3000;
    },
    refetchIntervalInBackground: false,
  });
}

POLLING PATTERN — Conformity Report
─────────────────────────────────────
// src/hooks/useConformityPoller.ts
import { useQuery } from '@tanstack/react-query';
import { queryKeys } from '@/lib/query-keys';
import { getConformityStatus } from '@/lib/api/notes';
import type { ConformityStatus } from '@/types';

const TERMINAL: ConformityStatus[] = ['DONE', 'FAILED'];

export function useConformityPoller(reportId: string) {
  return useQuery({
    queryKey: queryKeys.conformity.status(reportId),
    queryFn:  () => getConformityStatus(reportId),
    enabled:  !!reportId,
    refetchInterval: (query) => {
      const status = query.state.data?.status;
      return status && TERMINAL.includes(status) ? false : 4000;
    },
  });
}

MUTATION PATTERN — Login
─────────────────────────
import { useMutation } from '@tanstack/react-query';
import { login } from '@/lib/api/auth';
import { useAuthStore } from '@/stores/authStore';
import { decodeTokenSync } from '@/lib/token';
import { useRouter } from 'next/navigation';

export function useLogin() {
  const { setTokens } = useAuthStore();
  const router = useRouter();

  return useMutation({
    mutationFn: login,
    onSuccess: (data) => {
      const payload = decodeTokenSync(data.access);
      if (!payload) throw new Error('Invalid token received');
      setTokens(data.access, data.refresh, payload);
      // Role-based redirect
      const routes: Record<string, string> = {
        STUDENT:    '/student/dashboard',
        TEACHER:    '/teacher/dashboard',
        MAIN_ADMIN: '/admin/dashboard',
        SUB_ADMIN:  '/admin/dashboard',
        VISITOR:    '/visitor/dashboard',
      };
      router.push(routes[payload.role] ?? '/');
    },
  });
}

MUTATION PATTERN — Note Upload
───────────────────────────────
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { uploadNote } from '@/lib/api/notes';
import { queryKeys } from '@/lib/query-keys';
import { useRouter } from 'next/navigation';

export function useUploadNote() {
  const queryClient = useQueryClient();
  const router = useRouter();

  return useMutation({
    mutationFn: (formData: FormData) => uploadNote(formData),
    onSuccess: (data) => {
      // Invalidate the note list so it refetches
      queryClient.invalidateQueries({ queryKey: queryKeys.notes.all() });
      // Navigate to the status/review page
      router.push(`/student/notes/${data.note_id}/status`);
    },
  });
}

15. CRITICAL UX PATTERNS
================================================================================

PATTERN 1: ASYNC POLLING WITH VISUAL FEEDBACK
───────────────────────────────────────────────
Both the note pipeline and conformity analysis are async. Never make users
feel "stuck" — always show:
  1. A visual activity indicator (spinner, progress steps, animated icon)
  2. Estimated time ("This takes 15-30 seconds")
  3. What is happening ("Our OCR engine is reading your handwriting")
  4. Automatic redirect when done (no manual refresh needed)

Implement using useNotePoller / useConformityPoller hooks with refetchInterval.
Stop polling in TERMINAL states. Redirect via router.push in a useEffect that
watches the polled status.

PATTERN 2: MULTI-STEP UPLOAD UX
─────────────────────────────────
  Upload → (OCR pipeline) → OCR Review → (AI pipeline) → Summary Ready

The student MUST NOT be shown the AI summary before reviewing OCR.
Each transition is a separate screen/route to maintain clear mental model.
Use progress steps (breadcrumb-style) at the top of Note Processing and OCR
Review screens so the student knows where they are in the pipeline.

PATTERN 3: FILE UPLOAD PROGRESS
─────────────────────────────────
Always show an upload progress bar for file uploads.
Use Axios onUploadProgress:

  const response = await apiClient.post('/notes/upload/', formData, {
    headers: { 'Content-Type': 'multipart/form-data' },
    onUploadProgress: (evt) => {
      const percent = Math.round((evt.loaded * 100) / (evt.total ?? 1));
      setUploadProgress(percent);
    },
  });

Show a linear progress bar (shadcn Progress component) during upload.
Disable the upload button and show "Uploading..." text while in progress.

PATTERN 4: INLINE FORM VALIDATION
───────────────────────────────────
Use React Hook Form + Zod. Show errors inline under the relevant field,
not in a summary toast. This matches the DRF error response structure
where errors are field-keyed.

Map DRF 400 validation errors to React Hook Form:
  Object.entries(error.response.data).forEach(([field, messages]) => {
    form.setError(field as any, { message: (messages as string[])[0] });
  });

PATTERN 5: TOAST NOTIFICATIONS
────────────────────────────────
Use Sonner (shadcn's recommended toast library).
  - Success operations: brief green toast (3 seconds)
  - Error operations: red toast with error detail (5 seconds, no auto-dismiss)
  - Async operations (long): show progress toast while running, replace on complete

Toast placement: bottom-right on desktop, bottom-centre on mobile.

PATTERN 6: CONFIRM BEFORE DESTRUCTIVE ACTIONS
───────────────────────────────────────────────
Any DELETE or deactivation must show a ConfirmDialog (shadcn AlertDialog):
  - Title: specific action ("Delete this note?")
  - Description: consequence ("This cannot be undone. The file will be
    permanently removed from storage.")
  - Two buttons: [Cancel] (outline) + [Delete] (destructive red)

Never delete on single click. Always require confirmation.

PATTERN 7: EMPTY STATES
────────────────────────
Every list view needs an EmptyState component:
  - Icon (relevant to the list type)
  - Primary message ("No notes yet")
  - Secondary message ("Upload your first note to get started")
  - CTA button when appropriate ("Upload Note")

This prevents confusing blank pages when lists are empty.

PATTERN 8: OCR DRAFT SAVE
──────────────────────────
In the OCR Review screen, auto-save the text to localStorage every 10 seconds:
  key:   `ocr_draft_${noteId}`
  value: current textarea content

On load, check localStorage for a saved draft and restore it.
Clear the draft after successful confirmation.

This prevents data loss if the browser crashes or the student accidentally
navigates away while editing their OCR text.

PATTERN 9: PAGINATION
──────────────────────
Use cursor-style page navigation (Previous / Next + current page / total pages).
Read from PaginatedResponse: { count, next, previous, total_pages }.
PaginationControls component should show:
  "Showing {start}–{end} of {count} results"
  [← Previous] [1 2 3 ... 10] [Next →]

16. ERROR HANDLING STRATEGY
================================================================================

GLOBAL AXIOS ERROR HANDLING
─────────────────────────────
All API errors follow the RFC 7807 format:
  { error_code, detail, status_code, timestamp }

Or DRF validation errors:
  { field_name: ["message"], non_field_errors: ["message"] }

Create a helper to extract a user-friendly message:

  export function getErrorMessage(error: unknown): string {
    if (axios.isAxiosError(error)) {
      const data = error.response?.data;
      if (data?.detail)      return data.detail;
      if (data?.non_field_errors) return data.non_field_errors[0];
      // Fallback for DRF validation errors
      const firstField = Object.values(data ?? {})[0];
      if (Array.isArray(firstField)) return firstField[0];
    }
    return 'An unexpected error occurred. Please try again.';
  }

HTTP STATUS HANDLING MATRIX
─────────────────────────────
  400 Bad Request    → Show field-level validation errors inline in the form
  401 Unauthorized   → Trigger token refresh (handled in interceptor)
                       If refresh also fails → clear tokens → redirect to /
  403 Forbidden      → Toast: "You don't have permission to do this."
  404 Not Found      → Navigate to a 404 page or show EmptyState in place
  409 Conflict       → Show inline error in the relevant form (e.g. duplicate entry)
  429 Too Many Req.  → Toast: "Too many attempts. Please wait a minute and try again."
  500 Server Error   → Toast: "Server error. Our team has been notified."
                       Log to Sentry (recommended in production)

NETWORK ERRORS
───────────────
  if (error.code === 'ECONNABORTED') → "Request timed out. Check your connection."
  if (!error.response) → "Cannot reach the server. Check your internet connection."

FORM-LEVEL VS TOAST-LEVEL ERRORS
──────────────────────────────────
  Form validation errors (400):  Inline under each field
  Auth errors (401/403):          Toast notification
  Not found (404):                In-page error state (not a toast)
  Server errors (5xx):            Toast notification + optional retry button
  Conflicts (409):                Inline in the form or above the form

APPENDIX C: KNOWN BACKEND BEHAVIOURS & GOTCHAS
================================================================================

1. STUDENT AUTH: Students use 'username' (admission number), NOT email.
   The login form MUST change the identifier label to "Admission Number" for
   the Student tab. The API field is called "identifier" in both cases.

2. VISITOR SCHOOL_ID: Visitors have school_id = null in their JWT.
   All school-scoped API calls will fail for visitors.
   The visitor flow MUST NOT show any school-specific endpoints.

3. PUT vs PATCH for culture:
   The /admin/culture/ endpoint uses PUT (full upsert), not PATCH.
   Always send ALL culture fields — even empty strings. Do not send partial data.

4. DAILY CONTENT PATCH restriction:
   PATCH /admin/daily-content/{id}/ ONLY accepts 'body' and 'author'.
   Do NOT include content_type or display_date in PATCH requests — they are
   ignored. Show them as read-only in the edit form.

5. 409 CONFLICT on daily content creation:
   Each school can only have ONE entry per content_type per display_date.
   Show an inline error in the creation form, not a generic toast.

6. BULK CREATE response code:
   201 = all created successfully
   207 = partial success (some failed) — BOTH codes return the same response shape.
   Always check the 'failed' array in the response, regardless of status code.

7. NOTE STATUS after text upload:
   Text notes (note_type = 'text') skip OCR and immediately enter
   AWAITING_STUDENT_APPROVAL status. The upload response will show this status.
   Route directly to the OCR review screen (which shows the typed text for
   confirmation), not the processing status screen.

8. CONFORMITY report school scope:
   student_note_id must point to a note that is in the SAME school as the teacher.
   The backend enforces this via TenantManager. If you pass a note_id from
   another school, you'll get 404 NOT_FOUND (not 403 — this is by design to
   prevent enumeration).

9. VOICE NOTE OCR:
   Voice note OCR returns empty string (speech-to-text is Phase 3+).
   The OCR review screen MUST show a helpful message for voice notes:
   "Voice transcription is coming soon. Please manually type the content of
   your recording in the text area below."
   The confirm button should still work — the student types the content manually.

10. TOKEN REFRESH RACE CONDITION:
    Multiple simultaneous 401 responses must not trigger multiple refresh calls.
    The apiClient interceptor uses a queue pattern (failedQueue) to handle this.
    Ensure this logic is preserved exactly as written.

11. PASSWORD VALIDATION:
    The backend enforces Django's password validators:
    - Minimum 8 characters
    - Not entirely numeric
    - Not too similar to username/email
    Show these as inline form errors, not toasts.

12. ADMIN creating users:
    MAIN_ADMIN can create: STUDENT, TEACHER, SUB_ADMIN
    SUB_ADMIN can create:  STUDENT, TEACHER (NOT SUB_ADMIN)
    Hide the "Sub-Admin" role option in the create user form for SUB_ADMIN users.
    Read the current user's role from useAuthStore().role.
